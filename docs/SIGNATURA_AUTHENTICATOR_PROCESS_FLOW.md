# Signatura Authenticator Process Flow

**Status:** Implemented  
**Module:** Security → Authenticator  
**User route:** `/signatura/security/authenticator`

## Purpose

Signatura Authenticator generates temporary six-digit codes for applications approved by Signatura. It is an additional authentication option and does not replace or modify any existing QR login flow.

An authenticator result proves only that the person controlling a Signatura identity completed the required authentication steps. It does not communicate or store the user's application role, company, permissions, or module access. The consuming application remains responsible for authorization.

## Security boundaries

Before Signatura returns a code, all of the following must be true:

1. The user has a valid Signatura session.
2. The user account is active.
3. The identity satisfies Zero Trust Level 2.
4. The request comes from a currently trusted device.
5. The identity has an active enrollment for the requested application.
6. The application is active in the approved application registry.
7. If the application requires biometric verification, the session contains a recent successful passkey re-verification.

If trusted-device verification fails, the code is not returned or rendered.

## Components

| Component | Responsibility |
|---|---|
| `AuthenticatorApplication` | Approved application registry and application-level configuration |
| `AuthenticatorEnrollment` | Binds one Signatura identity to one approved application and encrypted secret |
| `AuthenticatorChallenge` | Short-lived, application-requested login challenge |
| Authenticator dashboard | Lists enrollments, performs biometric gating, and displays rotating codes |
| Authenticator service | Encrypts secrets, validates devices and clients, and generates codes |
| Consuming application | Requests challenges, submits codes, and makes its own authorization decisions |

The registry is application-neutral. ACCURA ERP is the initial approved application; HAVEN, WEKONNEK, PAYROLLPH, and other applications can use the same models and endpoints without application-specific code.

## Data model

### Approved application

```text
application_id
name
status
require_biometric
client_secret_hash
created_at
updated_at
```

`client_secret_hash` authenticates server-to-server requests. The plaintext application credential must not be stored in the database.

### Identity enrollment

```text
application_id
identity_id
secret_ciphertext
created_at
status
```

There can be only one enrollment for an `application_id` and `identity_id` pair. The authenticator secret is generated by Signatura and stored as an AES-256-GCM encrypted envelope. It is never returned through an API and is never sent to the consuming application.

### Login challenge

```text
token_hash
application_id
identity_id
status
expires_at
verified_at
created_at
```

Only the SHA-256 hash of the challenge token is persisted. A challenge starts as `pending`, expires after five minutes, and becomes `verified` after a successful code verification.

## Enrollment flow

```text
User opens Security → Authenticator
  ↓
Signatura validates session, account, Level 2, and trusted-device binding
  ↓
Dashboard loads active enrollments and approved applications
  ↓
User chooses Enroll for an approved application
  ↓
If require_biometric = true:
  Passkey / Face ID / fingerprint verification is completed
  ↓
Signatura generates a random 256-bit secret
  ↓
Secret is encrypted with AES-256-GCM
  ↓
Enrollment is stored for exactly one identity and one application
```

Enrollment does not exchange the secret with the application. The application verifies future codes by calling Signatura's server-side verification endpoint.

## Dashboard code flow

This flow supports a general-purpose application code when no login challenge token is present.

```text
User selects Generate Code
  ↓
Browser sends application ID and local trusted-device binding secret
  ↓
Server validates session, Level 2, trusted device, enrollment, and application
  ↓
Optional passkey/biometric gate
  ↓
Server decrypts the enrollment secret in memory
  ↓
Server generates a six-digit code for the current 30-second period
  ↓
Server records authenticator_viewed
  ↓
Dashboard displays code, countdown, and generation time
  ↓
At period expiry the dashboard requests a new code automatically
```

The response uses `Cache-Control: no-store`. Secret material is not included in the response.

## Code derivation

The server calculates the time counter as:

```text
counter = floor(unix_time_seconds / 30)
```

The HMAC message is scoped to the application, identity, optional login challenge, and current time counter:

```text
application_id : identity_id : challenge_id : counter
```

Signatura calculates HMAC-SHA-256 using the decrypted enrollment secret, performs dynamic truncation, and formats the result as six decimal digits.

Consequences of this design:

- Codes rotate every 30 seconds.
- Codes from one identity cannot authenticate another identity.
- Codes from one application cannot authenticate another application.
- A challenge-specific code cannot be reused for a different challenge.
- The consuming application never needs the authenticator secret.

## Application login challenge flow

```text
Consuming application login screen
  ↓
Application server requests a challenge from Signatura
  ↓
Signatura authenticates the application using its Bearer credential
  ↓
Signatura validates that the identity is enrolled
  ↓
Signatura creates a five-minute challenge and returns its one-time token
  ↓
User opens the Authenticator dashboard with that challenge token
  ↓
Trusted device and optional biometric checks run
  ↓
Signatura generates a code scoped to that challenge
  ↓
User enters the six-digit code in the application
  ↓
Application server submits challenge token and code to Signatura
  ↓
Signatura independently generates the expected code and compares it
  ↓
Success: challenge becomes verified and authenticator_login_success is logged
Failure: authenticator_login_failed is logged
  ↓
Application performs its own role and permission authorization
```

The challenge token can be supplied to the dashboard using:

```text
/signatura/security/authenticator?challenge=<challenge-token>
```

The token is bound server-side to its application and identity. Supplying it for another application or identity is rejected.

## API contracts

### List applications

`GET /api/authenticator/applications`

Browser requirements:

- Valid Signatura session cookie
- `x-device-binding` header containing the local trusted-device binding secret

Returns active identity enrollments and applications available for enrollment. It never returns authenticator secrets.

### Enroll identity

`POST /api/authenticator/enroll`

```json
{
  "applicationId": "accura-erp",
  "deviceBindingSecret": "device-local-secret"
}
```

Returns HTTP `428` when the application requires recent passkey verification and the session has not yet satisfied it.

### Generate or reveal code

`POST /api/authenticator/code`

General code request:

```json
{
  "applicationId": "accura-erp",
  "deviceBindingSecret": "device-local-secret"
}
```

Challenge-specific request:

```json
{
  "applicationId": "accura-erp",
  "deviceBindingSecret": "device-local-secret",
  "challengeToken": "short-lived-challenge-token"
}
```

Example response:

```json
{
  "applicationId": "accura-erp",
  "applicationName": "ACCURA ERP",
  "code": "483921",
  "generatedAt": "2026-07-11T12:00:08.000Z",
  "expiresIn": 22,
  "period": 30
}
```

### Create login challenge

`POST /api/authenticator/challenges`

Application server header:

```http
Authorization: Bearer <application-client-secret>
```

Request:

```json
{
  "applicationId": "accura-erp",
  "identityId": "SIGNATURA-UNIVERSAL-ID"
}
```

Response:

```json
{
  "challengeToken": "short-lived-challenge-token",
  "expiresAt": "2026-07-11T12:05:00.000Z"
}
```

This endpoint is separate from every QR challenge endpoint.

### Verify login code

`POST /api/authenticator/challenges/verify`

Application server header:

```http
Authorization: Bearer <application-client-secret>
```

Request:

```json
{
  "applicationId": "accura-erp",
  "challengeToken": "short-lived-challenge-token",
  "code": "483921"
}
```

Successful response:

```json
{
  "verified": true,
  "identityId": "internal-signatura-identity-id"
}
```

This response confirms identity ownership only. It is not an authorization token and contains no application role or permission data.

## Biometric and passkey gate

`require_biometric` is configured per application. When enabled:

1. The code endpoint checks for recent re-verification in the Signatura session.
2. If absent, it returns HTTP `428 Passkey verification required`.
3. The dashboard calls the existing WebAuthn re-authentication flow.
4. The platform authenticator may present Face ID, fingerprint, device PIN, or another passkey verification mechanism.
5. After successful re-verification, the dashboard retries the original request.

This reuses the existing passkey model and does not create a parallel biometric credential store.

## Audit trail

Authenticator events are written to both the security event and security audit logs through the existing authentication audit service.

| Event | Result | Recorded context |
|---|---|---|
| `authenticator_viewed` | success | Identity, application, trusted device, challenge when present, IP, user agent, timestamp |
| `authenticator_login_success` | success | Identity, application, challenge, IP, user agent, timestamp |
| `authenticator_login_failed` | failed | Identity, application, challenge, IP, user agent, timestamp |

Secrets, codes, challenge tokens, roles, company data, and permissions must never be written to audit details.

## Application onboarding

To add HAVEN, WEKONNEK, PAYROLLPH, or another application:

1. Generate a high-entropy application client secret outside the database.
2. Calculate its application client hash using the configured client pepper.
3. Add an `AuthenticatorApplication` row with a unique `application_id`, display name, status, biometric policy, and client-secret hash.
4. Deliver the plaintext client secret to the application's server through an approved secret-management channel.
5. Configure the application server to call only the authenticator challenge endpoints.
6. Never place the client secret in browser code.
7. Do not add application-specific conditions to the shared authenticator service.

## Environment configuration

| Variable | Requirement | Purpose |
|---|---|---|
| `AUTHENTICATOR_ENCRYPTION_KEY` | Required in production | Derives the AES-256-GCM key protecting enrollment secrets |
| `AUTHENTICATOR_CLIENT_PEPPER` | Strongly recommended | Keys the HMAC used to store application-client credential hashes |
| `SESSION_SECRET` | Existing platform requirement | Development fallback only; must not be treated as the production authenticator key |

Changing `AUTHENTICATOR_ENCRYPTION_KEY` without a secret re-encryption migration makes existing enrollments unreadable. Production rotation must therefore decrypt and re-encrypt enrollment envelopes under controlled operational tooling.

## Existing QR login isolation

Authenticator endpoints live only under `/api/authenticator/*`. The following existing capabilities remain separate and unchanged:

- QR Challenge
- QR Approval
- QR Login
- Trusted Device QR
- Desktop QR
- Mobile QR Approval

Authenticator is an additional login option. Implementations must not route authenticator requests through QR endpoints or reuse QR challenge records.

## Failure behavior

| Condition | Result |
|---|---|
| Missing or invalid session | `401 Authentication required` |
| Account inactive | `401 Authentication required` |
| Trust level below 2 | `403 Zero Trust Level 2 is required` |
| Device binding missing, revoked, or invalid | `403 Trusted device verification required` |
| Application not approved or inactive | `404 Application is not approved` |
| Enrollment missing or inactive | `404 Active enrollment not found` |
| Biometric verification required | `428 Passkey verification required` |
| Challenge invalid, consumed, or expired | `400 Challenge is invalid or expired` |
| Application credential invalid | `401 Invalid application credentials` |
| Code incorrect | `401 Invalid authenticator code` and failed-login audit event |

## Operational checklist

- Apply the authenticator Prisma migration.
- Configure `AUTHENTICATOR_ENCRYPTION_KEY` through the production secret manager.
- Configure `AUTHENTICATOR_CLIENT_PEPPER`.
- Provision a unique client credential for every approved application.
- Ensure application servers use HTTPS and keep their credentials server-side.
- Confirm trusted-device registration and passkey re-authentication work before enabling enrollment.
- Monitor failed verification events and add application/IP rate limits before broad public rollout.
- Define enrollment revocation and encryption-key rotation runbooks.
- Verify that no role, company, permission, or module-access attributes enter authenticator storage or responses.
